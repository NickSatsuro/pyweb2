Содержимое yml-файла

```yaml
name: Build and Deploy to GitHub Pages

on:
  push:
    branches:
      - main

permissions:
  contents: read
  pages: write
  id-token: write

concurrency:
  group: "pages"
  cancel-in-progress: false

jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - name: Checkout
        uses: actions/checkout@v4

      - name: Set up Python
        uses: actions/setup-python@v5
        with:
          python-version: '3.12'

      - name: Install dependencies
        run: |
          pip install -r requirements.txt

      - name: Set up Node JS
        uses: actions/setup-node@v4
        with:
          node-version: '20'
          cache: 'npm'

      - name: Install Node dependencies
        run: npm ci

      - name: Build Tailwind CSS
        run: npm run build:css

      - name: Build Mkdocs
        run: mkdocs build

      - name: Minify
        run: npm run minify

      - name: Validate
        run: npm run validate

      - name: Upload Artifact
        uses: actions/upload-pages-artifact@v3
        with:
          path: 'site' 

  deploy:
    needs: build
    runs-on: ubuntu-latest
    environment:
      name: github-pages
      url: ${{ steps.deployment.outputs.page_url }}
    steps:
      - name: Deploy to Github Pages
        id: deployment
        uses: actions/deploy-pages@v4
```

Этапы сборки

Процесс описывается в файле конфигурации рабочего процесса и инициируется автоматически при отправке изменений (push) в ветку main. Конвейер состоит из двух последовательных jobs: build (сборка) и deploy (развертывание).

Сборка выполняется на виртуальном окружении ubuntu-latest
Выполняется загрузка репозитория с помощью actions/checkout
Инициализируются среды выполнения Python версии 3.12 и Node.js версии 20 с кэшированием пакетов npm для ускорения повторных сборок
Устанавливаются Python-библиотеки согласно файлу requirements.txt
Устанавливаются Node.js-пакеты командой npm ci для обеспечения точного соответствия версий
Компилируются стили Tailwind CSS (npm run build:css).
Генерируется статический сайт документации с помощью MkDocs
Выполняется минификация ресурсов (npm run minify)
Проводится валидация сгенерированного HTML-кода (npm run validate)
Скомпилированная директория site загружается как артефакт сборки (pages-artifact)

Этап развертывания запускается только после успешного завершения этапа сборки.
Артефакт сборки развертывается в окружение github-pages с использованием официального действия actions/deploy-pages

## 1. Использование российских CDN для ускорения доставки контента

В условиях необходимости импортозамещения и обеспечения минимальных задержек (latency) внутри РФ российские CDN-провайдеры (Content Delivery Network) являются основным инструментом оптимизации сетевой доступности.

### Основные возможности

- **Минимизация RTT (Round Trip Time):**  
  Развитая сеть узлов фильтрации и кэширования в крупнейших точках обмена трафиком (MSK-IX и др.) позволяет доставлять контент из ближайшей к пользователю точки.

- **Защита от DDoS-атак:**  
  Интеграция с системами очистки трафика на уровнях L3, L4 и L7 (например, EdgeCenter, NGENIX).

- **Оптимизация контента «на лету»:**  
  Автоматическое сжатие изображений (WebP, AVIF) и Gzip/Brotli-сжатие текстовых файлов.

- **Поддержка отечественных сертификатов:**  
  Корректная работа с TLS-сертификатами НУЦ Минцифры РФ.

### Ключевые поставщики

- **EdgeCenter** — одна из самых широких сетей присутствия в СНГ.  
- **Cloud.ru / Selectel** — интегрированные решения (объектное хранилище + CDN).  
- **NGENIX** — специализация на защите и ускорении веб-ресурсов.

## 2. Возможности Gitverse для реализации CI/CD

Gitverse (платформа от Сбера) предоставляет среду для хостинга кода и автоматизации жизненного цикла разработки, сопоставимую по функциональности с международными аналогами.

### Технический инструментарий

- **Gitverse Actions** — автоматизация рабочих процессов на основе YAML-конфигураций (сборка, тестирование, деплой).
- **Собственные раннеры** — подключение изолированных вычислительных мощностей.
- **Интеграция с реестрами артефактов** — хранение Docker-образов и других бинарных файлов.

### Типовой сценарий CI/CD

1. **Trigger** — создание Pull Request или Push в основную ветку.  
2. **Build** — установка зависимостей и сборка проекта (например, `npm run build`).  
3. **Test** — запуск линтеров и unit-тестов.  
4. **Deploy** — передача собранных файлов на сервер или в объектное хранилище.

## 3. Варианты деплоя статического сайта в продакшен

Деплой статического сайта (HTML/CSS/JS/Assets) предполагает размещение файлов в среде, оптимизированной для быстрой обработки HTTP-запросов.

---

### Вариант А: Объектное хранилище (S3) + CDN

Наиболее масштабируемый метод.

Файлы загружаются в S3-совместимое хранилище, которое настраивается в режиме хостинга статического сайта.

**Инструменты:**
- AWS CLI  
- rclone  
- Terraform  

**Пример:**  
Использование S3 в Cloud.ru или Yandex Cloud с подключенным CDN.

---

### Вариант Б: Традиционный веб-сервер (Nginx) на VPS/VDS

Классический подход с полным контролем над серверной конфигурацией.

**Инструменты:**
- Nginx  
- SSH  
- rsync  
- CI/CD-раннеры  

**Пример:**  
Настройка `server`-блока в Nginx с указанием директории `root`, куда CI-скрипт копирует билд.

---

### Вариант В: Контейнеризация (Docker)

Статические файлы упаковываются в легковесный образ с Nginx и запускаются в контейнерной среде.

**Инструменты:**
- Docker  
- Docker Compose  
- Kubernetes  

**Пример:**  
Создание образа на базе `nginx:alpine`, копирование папки `dist` внутрь образа и запуск контейнера в Docker Swarm.
